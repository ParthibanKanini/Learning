# JVM Monitoring Report Documentation

This document explains each attribute in the JVM monitoring report generated by the application.

## Report Structure

### `logType`
Indicates the type of log entry being captured.
- **Values**: `"Initial"`, `"Periodic"`, `"Final"`
- **Purpose**: Identifies whether this is the first snapshot, an intermediate measurement, or the final report when the application completes

---

## `system` - Operating System Metrics

Operating system-level statistics about CPU, memory, and system resources.

### `cpuLoad`
CPU utilization metrics at both system and process level.

- **`systemPercent`**: Total CPU usage across the entire system (all processes), expressed as a percentage
`System Level` (systemPercent) refers to the entire computer - it measures CPU usage across all running processes and applications on your machine (browser, IDE, background services, other applications, etc.).

- **`processPercent`**: CPU usage by this specific JVM process, expressed as a percentage
`Process Level` (processPercent) refers to only this specific Java application - it measures how much CPU this particular JVM process is consuming.

**`Scenarios`**:
- `High system, low process`: Other applications are consuming CPU - your Java app isn't the bottleneck
- `High system, high process`: Your Java app is a major CPU consumer and may need optimization
- `Low system, low process`: System is idle, your app is running efficiently
- `Low system, high process`: Your app is the primary CPU consumer (e.g., doing intensive calculations)

### `availableProcessors`
Number of logical CPU cores available to the JVM for parallel processing.
Logical CPU cores are what the operating system sees as available processing units, which includes both physical cores and virtual cores created by technologies like Hyper-Threading (Intel) or SMT - Simultaneous Multithreading (AMD).

The JVM sees and can utilize all logical cores for parallel operations, but performance gain isn't linear when relying on hyper-threading (typically 20-30% improvement, not 100%).

- `Physical Cores`: Actual independent processing units on the CPU chip
- `Logical Cores`: Physical cores × threading technology

**`Scenarios`**:
- **No Hyper-Threading**: In 4 CPU physical core environment, if availableProcessors shows as 4 logical cores then, Ratio: 1:1
- **With Hyper-Threading (most common)**: In 4 CPU physical core envionment with Hyper-Threading enabled, availableProcessors would show 8 logical cores. Ratio: 1:2 (each physical core presents as 2 logical cores)

 NOTE: Windows: Task Manager → Performance tab → CPU

 **`How it helps`**:
- **Thread Pool Sizing**: Many applications set thread pool size based on availableProcessors
- **GC Threads**: Garbage collector parallelism is often based on available processors
- **Performance Tuning**: Understanding physical vs logical helps set realistic performance expectations (logical cores share execution resources)
- **Parallel Processing**: Fork/Join framework and parallel streams use this value

### `systemLoadAverage`
System load average over recent time periods. Indicates system workload and may show "Not available" on Windows systems.

Windows uses different metrics (CPU percentage, processor queue length) and doesn't calculate load average in the traditional Unix way. The JVM API returns -1 or "Not available" on Windows.

It is a Unix/Linux metric that measures the average number of processes waiting for CPU time over specific time periods. 

The load average shows **demand on the CPU** how many processes are either:
- Currently running on CPU
- Waiting in the run queue for CPU time
- In uninterruptible sleep (waiting for I/O)

If you have 8 logical cores (like your system):
- Load = 8.0: System is 100% utilized - all cores busy
- Load = 4.0: System is 50% utilized - half the cores busy
- Load = 12.0: System is overloaded - 50% more demand than capacity (processes waiting)
- Load = 2.0: System is underutilized - only 25% of cores busy

**`How it helps`**:
- **Context for CPU Usage** Your JVM shows 1.16% CPU usage
If load average is high (e.g., 6.0 on 8-core system), other processes are competing for CPU
Explains why your app might be slow despite low personal CPU usage
- **Capacity Planning** Load consistently near or above core count → need more CPU resources
Load well below core count → system has headroom for more workload
- **Performance Investigation**
  - High load + high JVM CPU: Your app is the bottleneck
  - High load + low JVM CPU: External processes affecting performance
  - Low load + slow app: Issue is not CPU-related (likely I/O, network, locks)
- **Scheduling Decisions**
High load average indicates CPU contention
JVM thread scheduling becomes less predictable. GC pauses may be longer due to CPU competition
- **Alerting Thresholds**
Set alerts when load > (cores × 0.7) sustained for 5+ minutes. Indicates system stress before users notice performance degradation


### `swapSpace`
Swap Space is disk storage used as virtual memory when physical RAM runs out. It's the OS's emergency overflow mechanism to prevent out-of-memory crashes.
When physical RAM is full, the operating system moves inactive memory pages from RAM to disk (swap file/partition). This frees up RAM for active processes.

- **`totalMB`**: Total swap space configured in megabytes
- **`freeMB`**: Available swap space in megabytes
- **`usedMB`**: Currently used swap space in megabytes

**`Performance Disaster`**
- RAM access: ~100 nanoseconds
- Disk (SSD) access: ~100,000 nanoseconds (1,000× slower)
- Disk (HDD) access: ~10,000,000 nanoseconds (100,000× slower)

If JVM heap gets swapped to disk:
- Garbage collection becomes extremely slow
Object access takes milliseconds instead of nanoseconds
- Application appears frozen or unresponsive
- Users experience severe lag

**`How to Protect JVM from Swap`**
- **Add More Physical RAM** 
- **Reduce JVM Heap Size** If you don't need 8 GB heap, reduce it: `java -Xmx4g -Xms4g`
- **Disable Swapping for JVM (Linux)** Lock JVM memory in RAM - prevents swapping
 `java -XX:+AlwaysPreTouch -XX:+UseTransparentHugePages`
- **Close/Kill Other Applications** 
- **Monitor GC Logs** If GC pauses suddenly increase from ~10ms to seconds, swapping is likely the cause

### `committedVirtualMemoryMB`
Amount of virtual memory guaranteed to be available to the process, in megabytes.

### `physicalMemory`
Physical RAM statistics for the entire system.
- **`totalMB`**: Total physical RAM installed in megabytes
- **`freeMB`**: Available physical RAM in megabytes
- **`usedMB`**: Currently used physical RAM in megabytes
- **`usagePercent`**: Percentage of physical RAM being used

### `uptime`
How long the JVM process has been running.

- **`uptimeMinutes`**: Uptime in minutes
- **`uptimeSeconds`**: Uptime in seconds (excluding complete minutes)
- **`uptimeMs`**: Precise uptime in milliseconds

## `heapMemory` - JVM Heap Memory

The heap is primary memory area where Java objects live.  Every time you use new to create an object, it's allocated on the heap. 

- Application Performance: If heap is too small, frequent garbage collections slow down your app
- Memory Efficiency: If heap is too large, you waste system resources
- GC Behavior: Heap size directly affects garbage collection frequency and duration
- OOM Prevention: Monitoring heap usage prevents OutOfMemoryError crashes

### Heap Summary
- **`maxMB`**: Maximum heap size the JVM can use (set by `-Xmx` JVM parameter). This is a Hard limit - JVM will never allocate more heap than this. If objects exceed this limit → OutOfMemoryError: Java heap space. 
- **`usedMB`**: Amount of heap currently occupied by live objects. Increases when you create objects (new)
Decreases when garbage collector reclaims dead objects. Shows actual memory consumption in real-time. 
- **`initMB`**: Initial heap size allocated at JVM startup (set by `-Xms` JVM parameter). Memory allocated at JVM startup
Best practice: Set -Xms equal to -Xmx to avoid dynamic heap resizing (which causes pauses).
- **`committedMB`**: Heap memory currently committed/reserved by the OS for the JVM TODO:...
- **`usagePercent`**: Percentage of maximum heap currently being used ((usedMB / maxMB) × 100). 

### `pools`
Memory pools within the heap. Structure varies by garbage collector (e.g., G1, CMS, Serial). The heap is divided into memory pools based on object lifecycles. This is called generational garbage collection - one of the JVM's most important performance optimizations.

Each pool contains:
- **`name`**: Name of the memory pool (e.g., "G1 Eden Space", "G1 Old Gen", "G1 Survivor Space")
- **`type`**: Always "Heap memory" for heap pools
- **`maxMB`**: Maximum size for this pool (may be "unlimited" for dynamic pools)
- **`usedMB`**: Current memory usage in this pool
- **`committedMB`**: Memory committed by OS for this pool
- **`usagePercent`**: Percentage of max being used, or "N/A" if max is unlimited

#### Common Pool Types:
- **Eden Space**: Where new objects are initially allocated.
   - You call new MyObject() → allocated in Eden
   - Eden fills up quickly (seconds to minutes)
   - Minor GC triggers → scans Eden for live objects
   - Dead objects discarded, survivors moved to Survivor Space
   - Small Eden → GC every second → high CPU overhead
   - Large Eden → GC every minute → better throughput, slightly higher pause
- **Survivor Space**: Objects that survived one Minor garbage collection cycle.
   - Acts as a "filter" - only objects that live long enough get promoted
   - Prevents Old Gen from filling with short-lived objects
   - Two survivor spaces allow efficient copying GC algorithm

- **Old Gen/Tenured**: Long-lived objects that survived multiple GC cycles
   - Contains application's "working set" - caches, singletons, long-lived data structures
   - Major GC is much slower than Minor GC (100-1000ms typical)
   - If Old Gen fills up → application can pause for seconds

### Tuning parameters
#### Heap sizing
`-Xms4g`          # Initial heap = 4 GB
`-Xmx4g`          # Max heap = 4 GB (match Xms for stability)

#### Young generation sizing (affects Eden/Survivor)
`-XX:NewRatio=2`  # Old Gen = 2× Young Gen
`-Xmn1g`          # Young Gen = 1 GB (explicit size)

#### Survivor tuning
`-XX:SurvivorRatio=8`   # Eden = 8× each Survivor space
`-XX:MaxTenuringThreshold=15`  # Survivals before promotion

### Best Practices
- Set -Xms = -Xmx: Prevents heap resizing overhead
- Monitor GC logs: Enable with -XX:+PrintGCDetails -Xlog:gc*:file=gc.log
- Young Gen = 25-30% of heap: Balance Minor vs Major GC
- Watch promotion rate: Old Gen should grow slowly
- Test under load: Production patterns differ from development
- Use appropriate GC: G1GC for heaps > 4GB, Serial for small apps
- Leave headroom: Don't allocate 100% of system RAM to JVM

The pool structure directly determines GC behavior:
**Minor GC**: Cleans Young Generation (Eden + Survivors) - Fast & Frequent
**Major GC**: Cleans Old Generation - Slow & Infrequent
**Full GC**: Cleans entire heap - Slowest (causes "Stop-the-World" pauses)
**Goal**: Keep most GC activity in Young Generation (Minor GC) because it's much faster than Major/Full GC.

## `nonHeapMemory` - Non-Heap Memory

Memory used by the JVM outside the heap for internal operations.

### Non-Heap Summary
- **`maxMB`**: Maximum non-heap memory (often "unlimited")
- **`usedMB`**: Current non-heap memory in use
- **`usagePercent`**: Percentage of max being used, or "N/A" if unlimited

### `pools`
Non-heap memory pools for JVM internal structures.

Each pool contains:
- **`name`**: Name of the non-heap pool
- **`type`**: Always "Non-heap memory"
- **`maxMB`**: Maximum size for this pool
- **`usedMB`**: Current usage
- **`committedMB`**: Memory committed by OS
- **`usagePercent`**: Percentage of max being used

#### Common Non-Heap Pools:
- **Metaspace**: Storage for class metadata (replaced PermGen in Java 8+)
- **Compressed Class Space**: Storage for compressed class pointers (when UseCompressedClassPointers is enabled)
- **CodeHeap 'non-nmethods'**: JVM internal code and buffers
- **CodeHeap 'profiled nmethods'**: JIT-compiled methods currently being profiled for optimization
- **CodeHeap 'non-profiled nmethods'**: Fully optimized JIT-compiled methods

---

## `classLoading` - Class Loading Statistics

Statistics about Java classes loaded into the JVM.

- **`loadedClassCount`**: Number of classes currently loaded in memory
- **`totalLoadedClassCount`**: Total number of classes loaded since JVM startup (includes unloaded classes)
- **`unloadedClassCount`**: Number of classes that were loaded and subsequently unloaded (garbage collected)

---

## `garbageCollection` - Garbage Collection Metrics

Information about garbage collection activity and performance.

### GC Summary
- **`totalCollectionCount`**: Total number of garbage collection cycles across all collectors
- **`totalCollectionTimeMs`**: Total time spent in garbage collection in milliseconds

### `collectors`
Statistics for each garbage collector active in the JVM.

Each collector contains:
- **`name`**: Name of the garbage collector (e.g., "G1 Young Generation", "G1 Old Generation", "G1 Concurrent GC")
- **`collectionCount`**: Number of times this collector has run
- **`collectionTimeMs`**: Total time spent in this collector in milliseconds
- **`avgCollectionTimeMs`**: Average time per collection in milliseconds, or "N/A" if no collections occurred

#### Common Collectors (G1 GC):
- **G1 Young Generation**: Minor GC - cleans up young generation (Eden and Survivor spaces)
- **G1 Old Generation**: Major/Full GC - cleans up old generation (long-lived objects)
- **G1 Concurrent GC**: Concurrent marking and cleanup (runs alongside application threads)

---

## `threads` - Thread Information

Statistics about threads in the JVM.

### `counts`
Thread count information.

- **`current`**: Number of live threads currently active
- **`peak`**: Peak number of live threads since JVM startup
- **`totalStarted`**: Total number of threads created since JVM startup
- **`daemon`**: Number of daemon threads (background threads that don't prevent JVM shutdown)

### `states`
Distribution of threads across different execution states.

- **`new`**: Threads created but not yet started
- **`runnable`**: Threads executing or ready to execute (has CPU or waiting for CPU)
- **`blocked`**: Threads blocked waiting to acquire a monitor lock
- **`waiting`**: Threads waiting indefinitely for another thread's action (e.g., `Object.wait()`)
- **`timedWaiting`**: Threads waiting for a specific time period (e.g., `Thread.sleep()`, `Object.wait(timeout)`)
- **`terminated`**: Threads that have completed execution

### `deadlockedThreadCount`
Number of threads currently in a deadlock state. A non-zero value indicates a serious problem.

---

## `jvmInfo` - JVM Information

Metadata about the Java Virtual Machine instance.

- **`jvmName`**: Name of the JVM implementation (e.g., "Java HotSpot(TM) 64-Bit Server VM")
- **`jvmVersion`**: Version string of the JVM
- **`jvmVendor`**: Vendor/provider of the JVM (e.g., "Oracle Corporation", "Eclipse Adoptium")
- **`pid`**: Operating system process ID for this JVM instance
- **`startTime`**: JVM start time as Unix timestamp in milliseconds (milliseconds since January 1, 1970 UTC)

---

## `timestamp`
Unix timestamp in milliseconds when this report snapshot was captured.

---

## Usage Notes

### Memory Units
All memory values are reported in **megabytes (MB)** unless otherwise specified.

### Unlimited Values
Some memory pools show `"unlimited"` for `maxMB` when they can grow dynamically without a preset maximum.

### N/A Values
`"N/A"` appears for calculated values (like `usagePercent`) when the maximum is unlimited, making percentage calculation impossible.

### Interpretation Tips
- **Low heap usage %**: Application has sufficient memory headroom
- **High GC count/time**: May indicate memory pressure or excessive object creation
- **Blocked threads**: Potential contention issues or deadlocks
- **High swap usage**: System may be thrashing; consider adding more RAM
- **Zero deadlocks**: Healthy threading behavior
